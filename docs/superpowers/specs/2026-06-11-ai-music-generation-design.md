# AI Music Generation Integration Design

## Hardware Context

**Target Hardware:**
- RAM: 64GB
- GPU: NVIDIA GeForce RTX 5090 Laptop GPU (24GB VRAM reported by `nvidia-smi`)
- OS: Windows 11

This configuration supports current open-weight music generation models well, but it should be treated as a high-end laptop GPU target rather than a desktop RTX 5090 32GB target. Favor models that fit in 24GB VRAM with one active generation job.

## Model Recommendations

### Primary Recommendation: Stable Audio 3 Small Music

**Recommended model:** `stabilityai/stable-audio-3-small-music`

**Why this should be the default for pure instrumental generation:**
- Latest Stable Audio 3 music-specialized small model.
- Better fit for this app's use case: pure instrumental background music, samples, loops, and sound design.
- Official Stable Audio 3 Small-Music model is listed as 433M parameters, with a small local footprint.
- Official model examples support 120 second music generation.
- Stable Audio 3 / Stable Audio Tools provides the intended inference path.

**Default product stance:**
- Use this for normal "generate pure music" requests.
- Default duration should be 30 seconds for responsiveness.
- Allow 5-120 seconds for Stable Audio 3 Small Music.
- Build loop/extend workflows at the app layer instead of requiring a huge lyrics/song model.
- Do not include lyrics fields in the default UI.

**Input Format:**
```python
# Text-only generation
{
  "prompt": "upbeat electronic dance music with synthesizers",
  "duration": 30,
  "job_id": "uuid",
  "output_dir": "/app-data/music"
}
```

**Output:**
- Format: WAV generated by the Stable Audio 3 Small Music service
- Encoding: Store WAV directly for app playback/history. Do not transcode to MP3 in this workflow.

**Expected fit on RTX 5090 Laptop 24GB:**
- Stable Audio 3 Small Music should run comfortably on this GPU.
- Use bf16/fp16 where supported.
- Keep max concurrent generations at 1 to avoid VRAM fragmentation.

## Architecture Design

### System Overview

```
Tauri frontend
  -> Tauri Rust commands
  -> Python FastAPI service
  -> Stable Audio 3 Small Music on GPU
  -> WAV files in app data
  -> SQLite generation history
```

### Component Breakdown

**1. Python Generation Service (Standalone Process)**

Responsibilities:
- Load Stable Audio 3 Small Music model on startup
- Accept generation requests via HTTP API
- Run inference on GPU
- Save generated audio directly as WAV
- Return file path or stream audio

**Why separate process:**
- Rust doesn't have mature PyTorch bindings
- Python has better AI ecosystem
- Isolates GPU crashes from main app
- Easier model updates

**API Specification:**
```python
# POST /generate
Request:
{
  "prompt": "string",
  "duration": 5-120,
  "model_version": "stable-audio-3-small-music",
  "job_id": "uuid"  # provided by Tauri
}

Response:
{
  "job_id": "uuid",
  "status": "queued" | "processing" | "completed" | "failed",
  "progress": 0-100,
  "output_path": "/path/to/generated.wav",  # when completed
  "error": "string | null"
}

# GET /job/{job_id}
Response: Same as above

# DELETE /job/{job_id}
Response: { "cancelled": true }

# GET /health
Response: { "status": "ready", "model_loaded": true, "gpu_available": true }
```

**2. Tauri Backend (Rust)**

New modules:
- `src-tauri/src/music_generation.rs` - Generation job management
- `src-tauri/src/music_history.rs` - History CRUD operations
- `src-tauri/src/python_client.rs` - HTTP client for Python service

Database schema:

CREATE TABLE music_history (

**User Isolation:**
- Each user sees only their own generated music
- Deleting a user cascades to their music history
- File cleanup removes orphaned files when user is deleted

## File Storage Strategy

**Generated files location:**
```
%APPDATA%/dev.tauri-eeg.app/music/
  |-- gen_20260611_120345_a1b2c3.wav
  |-- gen_20260611_120512_d4e5f6.wav
  `-- ...
```

**Naming convention:**
- `gen_{YYYYMMDD}_{HHMMSS}_{short_hash}.wav`
- Short hash from job_id (first 6 chars)
- Prevents collisions, sortable by time

**Cleanup policy:**
- Keep last 50 generations
- Delete files older than 30 days (configurable)
- Never auto-delete favorited items
- Show storage usage in UI

## Security Considerations

**Prompt validation:**
- Max length: 500 characters
- Strip filesystem paths and shell commands
- Rate limit: 1 generation per 10 seconds per user

**File access:**
- Never expose Python service port to network (localhost only)
- Validate all file paths come from app data directory
- Sanitize filenames before saving

**Resource limits:**
- Max concurrent generations: 1 (sequential processing)
- Max duration: 120 seconds for Stable Audio 3 Small Music
- Timeout: 60 seconds per generation

## Python Service Implementation

**Recommended stack:**
- FastAPI (async HTTP server)
- Official Stable Audio 3 library from `https://github.com/Stability-AI/stable-audio-3`
- torchaudio for WAV output through the model stack
- uvicorn (ASGI server)

**Installation:**
```bash
uv init --python 3.11
uv add fastapi==0.115.0 "uvicorn[standard]==0.30.0" pydantic==2.9.0
uv add "stable-audio-3 @ git+https://github.com/Stability-AI/stable-audio-3.git"
uv sync
```

Before first run, accept the Hugging Face terms for `stabilityai/stable-audio-3-small-music` and authenticate with Hugging Face locally if required.

**Service sketch:**
```python
from fastapi import FastAPI

app = FastAPI()
model = None

@app.on_event("startup")
async def load_model():
    global model
    model = StableAudioModel.from_pretrained("small-music").to(device)

@app.post("/generate")
async def generate_music(request: GenerateRequest):
    audio = await model.generate(
        prompt=request.prompt,
        duration=request.duration,
    )
    output_path = save_wav(audio, request.job_id, request.output_dir)
    return {"job_id": request.job_id, "output_path": output_path}
```

## Deployment Strategy

**Automatic Service Management (Recommended):**

Tauri automatically manages the Python service lifecycle:

1. **On first generation request**: Check if service is running
2. **If not running**: Start Python subprocess automatically
3. **Health check**: Wait for service to be ready (max 30s)
4. **Process management**: Keep handle in global state
5. **On app exit**: Send graceful shutdown signal to Python process

**Implementation approach:**
```rust
// Global state holds Python process handle
struct MusicServiceState {
    process: Option<Child>,
    is_running: bool,
}

// Auto-start on first generate_music call
async fn ensure_service_running(state: &mut MusicServiceState) -> Result<(), String> {
    if state.is_running {
        return Ok(());
    }
    
    // Start: cd music-service && uv run python server.py
    let child = Command::new("uv")
        .current_dir("music-service")
        .args(&["run", "python", "server.py"])
        .spawn()?;
    
    state.process = Some(child);
    state.is_running = true;
    
    // Wait for health check
    retry_health_check(30).await?;
    
    Ok(())
}
```

**User experience:**
- User clicks "Generate" 鈫?Service auto-starts if needed
- First generation takes ~5-10s longer (model loading)
- Subsequent generations are fast
- No manual terminal commands required

**Fallback for development:**
- If auto-start fails, show error with manual start instructions
- User can still start service manually if needed

## Performance Targets

With Stable Audio 3 Small Music on RTX 5090 Laptop 24GB:
- Generation time: target < 20 seconds for 30s audio after model warmup
- UI responsiveness: No blocking during generation
- Memory usage: fit within 24GB GPU VRAM with one active job
- Storage: roughly 5MB per 30 seconds for 44.1kHz stereo 16-bit WAV. This is acceptable for the app and avoids transcoding time/dependencies.
